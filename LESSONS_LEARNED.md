## 2. 浮動小数点の罠

### 何が起きたか
スコア合計が `89.70000000000003` のように表示された。

### 原因
JavaScriptの浮動小数点演算。`0.1 + 0.2 = 0.30000000000000004` と同じ問題。
`.reduce()` でスコアを加算していくと誤差が蓄積する。

### 解決策
```typescript
// ダメ
const total = items.reduce((sum, item) => sum + item.score, 0)

// 正解: 最後にround
const total = Math.round(
  items.reduce((sum, item) => sum + (item.score || 0), 0) * 10
) / 10
```

### 教訓
- **金額・スコアなど数値を合計する箇所は全て `Math.round` で丸めろ**
- 表示時は `.toFixed(1)` を統一的に使う
- 計算箇所は1つのページに複数あることが多い（一覧表示・詳細表示・PDF出力）。**全箇所を漏れなく対応**すること
- テンプレートの配点で `step="0.5"` のような小数入力を許可する場合は特に注意

## 3. PDF出力

### 踏んだ地雷一覧

#### 3-1. 真っ白なPDFが出力される
**原因**: `html2canvas` がDOMに追加される前の要素をキャプチャしようとした
**対策**: 必ず `document.body.appendChild(tempDiv)` してからキャプチャ。`finally` で確実に `removeChild` する

#### 3-2. 日本語が文字化けする
**原因**: html2canvasはシステムフォントに依存する
**対策**: CSSで `font-family` を明示的に指定
```css
font-family: 'Noto Sans JP', 'Hiragino Sans', 'Hiragino Kaku Gothic ProN', Meiryo, sans-serif;
```

#### 3-3. 項目数が多いとページに収まらない
**対策**: 項目数に応じてフォントサイズを動的に調整
```typescript
const fontSize = itemCount > 40 ? '8px' : itemCount > 30 ? '9px' : '10px'
```

#### 3-4. 型定義にフィールドを追加し忘れる
**原因**: PDFデータの型、画面の型、DBからのマッピングの3箇所を全て更新する必要がある
**対策**: データの流れを把握しておく
```
DB (select *) → マッピング関数 → 画面の型 → buildPDFData → PDF型 → HTML生成
```
**1箇所でも漏れると表示されない。新フィールド追加時はこの全経路を確認。**

#### 3-5. デプロイ後にPDFが変わらない
**原因**: PDFはクライアントサイドJSで生成される。ブラウザキャッシュで古いコードが動く
**対策**: ユーザーに**ハードリロード（Cmd+Shift+R）**を伝える

### PDF設計の教訓
- PDFのレイアウトはHTMLで組んで `html2canvas` → `jsPDF` が楽だが、ページ分割の制御が難しい
- 1ページに収める `fitToPage` モードと、複数ページにスライスするモードを両方用意しておく
- ファイル名は日本語で分かりやすく（`評価レポート_山田太郎_2024上期_20240315.pdf`）

## 4. 権限・閲覧制御

### 今回の権限体系
| | 管理者 | MG | 店長 | スタッフ |
|---|---|---|---|---|
| ユーザー管理 | ○ | ○ | × | × |
| テンプレート | ○ | ○ | × | × |
| 評価期間 | ○ | ○ | ○ | × |
| 評価閲覧 | 全員 | 管轄店舗 | 自部署 | 自分のみ |
| 総評コメント | ○ | ○ | × | × |
| 項目コメント | ○ | ○ | ○ | 確定final以外 |
| ランキング | 全員 | 管轄店舗 | × | × |

### 踏んだ地雷

#### 4-1. 権限チェックが散らばる
各ページで `if (user.role === 'admin')` を繰り返し書いた結果、修正漏れが発生した。

**次回の対策**: 権限チェックを1箇所に集約
```typescript
// lib/permissions.ts
export const canViewEvaluation = (user: User, evaluation: Evaluation) => { ... }
export const canEditEvaluation = (user: User, evaluation: Evaluation) => { ... }
export const canViewOverallComment = (user: User) => { ... }
```

#### 4-2. 「見える」と「編集できる」は別
結果ページで「見える」が、評価ページでは「編集できる」判定が必要。
さらに「確定済み期間は誰も編集できない」という横断的ルールもある。

**教訓**: 権限マトリクスを最初にドキュメント化してから実装する

#### 4-3. スタッフに見せる/見せないの境界
「確定」ボタンを押すとスタッフに最終評価が見えるようになるが、総評コメントと項目コメントは非表示にする必要があった。
データフェッチ後にフィルタリングする処理を入れた。

**教訓**: 「このデータをこの役割に見せていいか？」を全フィールドに対して考える

## 5. データ保存の競合

### 何が起きたか
評価入力で、グレード選択とコメント入力が同時に起きると片方が消える。

### 原因
自動保存が走るタイミングで、古いstateを元にupsertしてしまう。
グレード変更 → 保存開始 → コメント変更 → グレード保存完了（コメントが古い値で上書き）

### 解決策
変更されたフィールドだけを更新する。state全体をupsertしない。

### 教訓
- **自動保存は楽だが競合に注意**。変更のあったフィールドだけを保存する
- デバウンス（300ms程度）を入れて連続更新を抑制する
- 保存中フラグを持ち、二重保存を防ぐ
- コメント入力は `onBlur` で保存するのが安全（`onChange` だと毎文字保存になる）

## 6. Supabaseとの付き合い方

### 良かった点
- `select('*')` で全カラム取得は楽。型安全にしたい場合は必要カラムを明示
- upsert の `onConflict` が便利。評価スコアの初回作成・更新を1つのクエリで処理できる
- RLS（Row Level Security）を使えばDB層でアクセス制御ができる（が今回は未使用）

### 次回やるべきこと
- **RLSを最初から設定する**: アプリ側の権限チェックだけだと、Supabaseのクライアントキーがバレた場合にデータ丸見え
- **型生成を使う**: `supabase gen types typescript` でDB型を自動生成。手動の型定義と実際のDBのズレを防げる
- **`select('*')` の罠**: テーブルにカラムが増えると不要なデータも取得する。パフォーマンスに影響が出たら必要なカラムだけselectする

### N+1問題
評価ごとにスコアを個別取得すると遅くなる。一括取得してMapで参照するパターンを使う。
今回はevaluationごとにスコア取得が残っている箇所がある。データ量が増えたら要改善。

## 7. 状態管理の爆発

### 何が起きたか
`evaluations/page.tsx` が2000行超え。state変数が30個以上。

### なぜこうなったか
- 評価一覧、評価入力フォーム、提出済み一覧、参照評価表示が全て1ページ
- フィルタリング、ソート、モーダル、自動保存がそれぞれstate持ち

### 次回の対策
- **コンポーネント分割を早い段階でやる**
  ```
  evaluations/
    page.tsx              # メインレイアウト
    EvaluationList.tsx     # 一覧
    EvaluationForm.tsx     # 入力フォーム
    SubmittedList.tsx      # 提出済み一覧
    ReferencePanel.tsx     # 参照評価
  ```
- **カスタムフックに抽出**
  ```typescript
  const { evaluations, filters, setFilters } = useEvaluationList()
  const { currentEval, saveItem, submitEval } = useEvaluationForm(evalId)
  ```
- 1ファイル500行を超えたら分割を検討する

## 8. デプロイ・Vercel

### 踏んだ地雷

#### 8-1. ビルド成功なのにデプロイ失敗
**原因**: Vercelのプラットフォーム障害（`We encountered an internal error`）
**対策**: ビルドログを確認して `Deploying outputs...` で止まっていたらVercel側の問題。Vercel Statusページを確認

#### 8-2. ローカルビルドを通してからpush
型エラーや未使用import等でVercel上でビルドが落ちると時間の無駄。
**`npm run build` をローカルで通す習慣をつける。**

#### 8-3. クライアントサイドのキャッシュ
Next.jsのクライアントコンポーネントはブラウザにキャッシュされる。
デプロイ直後に「変更が反映されない」と言われたら**ハードリロード**を案内する。

## 9. UI/UXで後から効いてくるやつ

### 9-1. レスポンシブは最初から考える
後からスマホ対応すると全テーブル・全ダイアログの修正が必要になる。
**最初から `md:` ブレークポイントを意識して組む。**

### 9-2. 一覧のソート・フィルターは必須
ユーザーが増えると「自分の評価が見つからない」問題が発生する。
**部署フィルター、ステータスフィルター、名前検索は初期実装に含める。**

### 9-3. 進捗表示があると安心感が違う
評価項目が50個あると、どこまで入力したか分からなくなる。
**プログレスバー + 未入力項目へのジャンプ機能は早めに入れる。**

### 9-4. 確認ダイアログを入れるべき場所
- 評価の提出（取り消し可能だが心理的に大きい操作）
- 期間の確定（取り消し不可）
- 評価の削除
- 一括操作（一括確定など）

### 9-5. 数値の表示形式は統一する
`89.7` と `89.70` と `89.70000000000003` が混在すると不信感を与える。
**`.toFixed(1)` を全箇所で統一。** 表示用のユーティリティ関数を1つ作る:
```typescript
export const formatScore = (score: number): string => score.toFixed(1)
```

## 10. 設計を最初に決めておくべきだったこと

### 10-1. 評価フローの状態遷移
```
pending → in_progress → submitted → confirmed
                                      ↓
                                    (locked by period completion)
```
ステータスの種類と遷移ルールを最初に決めておく。後から `confirmed` を追加したため、全ページに影響が出た。

### 10-2. 権限マトリクス
上で書いた通り。**最初にスプレッドシートで権限表を作ってから実装する。**

### 10-3. テーブル設計のポイント
- `evaluations` テーブルに `stage` カラムを持たせたのは正解。期間x被評価者xステージでユニーク制約
- `evaluation_scores` を別テーブルにしたのも正解。項目数が可変なため
- `overall_grade` と `final_decision` は後追いで追加した。最初から最終評価用のカラムを想定しておく

### 10-4. ファイル名・命名規則
- ページファイルは `page.tsx` 固定（App Router）だが、コンポーネントは機能名で分ける
- 型定義は `types/` に集約するか、使う場所の近くに置くかを決めておく
- Supabaseのテーブル名はsnake_case、TypeScriptの型はPascalCase。変換ルールを統一

## 11. SaaS/Webアプリ設計の一般Tips

今回のプロジェクトに限らず、SaaSやWebアプリ全般で使える設計指針。

---

### 11-1. 最初に決める5つのこと

新しいプロジェクトを始める前に、コードを1行も書かずにこの5つを決める:

1. **誰が使うか（ユーザーの種類）**: 管理者/一般ユーザー/閲覧者 etc.
2. **何ができるか（機能一覧）**: CRUD単位で洗い出す
3. **誰が何を見れるか（権限マトリクス）**: 全組み合わせを表にする
4. **データの状態遷移**: 下書き→公開→アーカイブ etc.
5. **画面一覧とURL設計**: `/dashboard/users`, `/dashboard/settings` etc.

これを決めずに作り始めると、後から「あ、この画面にこの権限チェック入れてなかった」が頻発する。

---

### 11-2. DB設計の鉄則

#### テーブルを正規化しすぎない
- JOINが多くなるとパフォーマンス悪化
- 「いつも一緒に取得するデータ」は同じテーブルに入れてOK
- ただしリレーションが明確なもの（ユーザーと評価スコアなど）は分ける

#### 「後から追加するカラム」を想定する
- `metadata` や `options` のようなJSONBカラムを1つ持たせておくと、軽微な追加でマイグレーション不要
- ただし検索対象にするカラムは正規カラムにする

#### ソフトデリートを使う
```sql
-- 物理削除ではなく
DELETE FROM users WHERE id = '...'

-- ソフトデリート
UPDATE users SET deleted_at = NOW() WHERE id = '...'
```
- ユーザーが「やっぱり戻して」と言ったときに対応できる
- 関連データの整合性も保てる

#### タイムスタンプは必ず入れる
- `created_at`, `updated_at` は全テーブルに必須
- 「いつ作られたか」「いつ更新されたか」はデバッグにも分析にも必要

---

### 11-3. API/データフェッチ設計

#### フロントからDBを直接叩く場合（Supabase等）
- RLSで必ずアクセス制御する
- `select('*')` は開発初期だけ。本番ではカラム指定を推奨
- 複数テーブルのデータを使う場合、`Promise.all` で並列取得

#### APIルートを立てる場合（Next.js API Routes等）
- バリデーションは入口で全てやる（Zodが便利）
- エラーレスポンスの形式を統一する
```typescript
// 統一エラー形式
{ error: true, message: "具体的なエラー内容", code: "INVALID_INPUT" }
```
- 成功レスポンスも統一
```typescript
{ data: {...}, message: "OK" }
```

#### キャッシュ戦略
- 頻繁に変わらないデータ（ユーザー一覧、テンプレート等）はキャッシュする
- `stale-while-revalidate` パターン: 古いデータを先に表示して、裏で更新
- SWRやTanStack Queryを使うと楽

---

### 11-4. エラーハンドリングの3層構造

```
① UI層:       ユーザーに見せるエラーメッセージ（日本語、分かりやすく）
② ロジック層:  catchしてログ出力 + UI層に伝播
③ データ層:    DB/APIのエラーをcatch
```

#### やりがちなミス
- `catch (e) {}` で握りつぶす → 原因不明のバグになる
- 技術的エラーをそのまま表示 → ユーザーが混乱する
- 全てalert() → UXが悪い

#### 正しい対応
```typescript
try {
  const { error } = await supabase.from('users').insert(data)
  if (error) throw error
} catch (err) {
  console.error('ユーザー作成エラー:', err)  // 開発者向けログ
  setError('ユーザーの作成に失敗しました。もう一度お試しください。')  // ユーザー向け
}
```

---

### 11-5. フォーム設計のベストプラクティス

#### バリデーションは3箇所でやる
1. **HTML属性**: `required`, `maxLength`, `pattern` → 即座にフィードバック
2. **フロント側**: Zod + React Hook Form → 送信前にチェック
3. **サーバー側/DB**: NOT NULL制約、CHECK制約 → 最後の砦

#### 送信ボタンの状態管理
```
未入力     → disabled（グレーアウト）
入力中     → enabled
送信中     → loading（スピナー表示 + disabled）
成功       → 成功メッセージ or リダイレクト
失敗       → エラーメッセージ + ボタン再活性化
```

#### 離脱防止
フォーム入力中にページを離れようとしたら警告を出す:
```typescript
useEffect(() => {
  const handler = (e: BeforeUnloadEvent) => {
    if (hasUnsavedChanges) e.preventDefault()
  }
  window.addEventListener('beforeunload', handler)
  return () => window.removeEventListener('beforeunload', handler)
}, [hasUnsavedChanges])
```

### 11-6. ローディングとエンプティステート

全ての画面に3つの状態を用意する:

1. **ローディング中**: スケルトンUIまたはスピナー
2. **データあり**: メインの表示
3. **データなし**: 「まだデータがありません」+ 次のアクションを案内

```typescript
if (loading) return <Skeleton />
if (data.length === 0) return <EmptyState message="評価がありません" action="期間管理で割り当ててください" />
return <DataTable data={data} />
```

「真っ白な画面」が一番不安を与える。必ず何かを表示する。

### 11-7. URL設計とディープリンク

#### URLを機能のIDにする

- ブックマーク可能になる
- 他の人にリンクを共有できる
- ブラウザの戻る/進むが正しく動く

#### URLSearchParamsを活用
```typescript
const searchParams = useSearchParams()

```

### 11-9. 通知・メール設計

SaaSで必ず必要になる機能:

- **何を通知するか**: 評価依頼、提出完了、確定通知 etc.
- **どう通知するか**: メール / アプリ内通知 / Slack連携
- **頻度制御**: 即時 / 日次ダイジェスト / 週次サマリー

後から追加すると全ページに手を入れることになるので、イベント発行の仕組みだけ最初に作っておく:
```typescript
// lib/events.ts
export const emitEvent = (type: string, payload: any) => {
  // 将来ここにメール送信やSlack通知を追加
  console.log(`[Event] ${type}`, payload)
}

// 使う側
emitEvent('evaluation.submitted', { evaluationId, evaluatorId })
```

---

### 11-10. テスト戦略

全部テストする必要はないが、最低限これだけは:

- **権限チェック**: 各ロールで見えるもの/見えないものをテスト
- **状態遷移**: 正常系 + 不正な遷移（submitted → pending に戻せないこと等）
- **計算ロジック**: スコア計算、ランキングの順位付け
- **E2Eテスト**: ログイン → 評価入力 → 提出 の一連の流れ

フロントのUIテストよりロジックのテストを優先する。
